

# 一、项目概述

xxx项目（Data Agent） 是一款**面向数据仓库**的自然语言查询 智能体系统（**NL2SQL**）。

用户只需用中文提问，系统便能自动理解问题语义、召回相关**元数据**、生成SQL校验 SQL，最终执行数仓查询SQL，返回结构化查询结果——全程无需手写 SQL，大幅降低数据分析门槛。

数据库设计方面分两部分：

**数据仓库**采用经典**星型模型**，存不同领域业务数据，比如围绕订单事实表构建：

| 表名         | 类型   | 说明                                                         |
| ------------ | ------ | ------------------------------------------------------------ |
| fact_order   | 事实表 | 订单核心数据，含 order_amount（GMV）、order_quantity（销量） |
| dim_customer | 维度表 | 客户属性：姓名、性别、会员等级                               |
| dim_product  | 维度表 | 商品属性：品类、品牌、商品名称                               |
| dim_region   | 维度表 | 地区属性：省份、大区（华东/华南等）                          |
| dim_date     | 维度表 | 时间维度：年、季度、月、日                                   |



**元数据库：**

| 表名                      | 说明                                                         |
| ------------------------- | ------------------------------------------------------------ |
| table_info表信息          | 存储数仓中维度表 / 事实表的基础元数据，标识表类型、描述表含义 |
| column_info字段信息       | 存储每张表的字段级元数据，包含字段类型、数据示例、别名、所属表关联等 |
| metric_info指标信息       | 存储数仓核心指标的元数据，定义指标含义、关联列、业务别名     |
| column_metric字段指标信息 | 字段和指标的**多对多关联中间表**，建立列与指标的映射关系     |

# 三、项目主线业务流程

系统核心是采用 **LangGraph 编排的多节点** AI Agent 工作流。用户的一次自然语言查询，将依次经历六个阶段共计12个节点



## 阶段一：语义解析（关键字提取）

extract_keywords — 关键字抽取

◦ **使用 jieba  对用户问题分词，提取名词、动词、专有名词等核心词性**

◦ 将原始问题本身与分词结果合并去重，形成初始关键字集合

 

## 阶段二：多路并行召回（RAG 检索增强）

extract_keywords 之后，三条支路并行执行，互不阻塞： 把跟问题有关的字段信息、指标信息、以及字段的取值获取到，

recall_column — **字段语义召回**

- 大模型将用户问题扩展为「字段名列表」（如：当日，品类、销售金额、地区）


- 对每个关键词向量化，查询 Qdrant 向量库，基于余弦相似度（阈值 0.6）召回相关字段信息

recall_metric — **指标语义召回**

- 大模型将问题扩展为「指标概念词」（如：GMV、成交额、订单总额）


- 向量化后查询 Qdrant 中的指标向量集合，召回候选指标

recall_value — **字段取值召回**

- LLM 将问题扩展为「字段枚举值」（如：华南、铂金、手机数码）


- 查询 Elasticsearch 全文索引，使用 IK 中文分词器，采用全文查询匹配字段取值




## 阶段三：信息合并（merge_retrieved_info）

• 汇聚三路召回结果合并，以字段 ID 为 Key 去重合并。因为拿到字段就可以推出表

• 补全指标关联字段：若指标对应字段不在召回字段集中，从 Meta MySQL 查询并且补充字段信息

• 将 ES 召回的字段值注入对应字段的 examples 列表，增强生成 SQL 的过滤条件准确性

• 手动补全所有表的主键与外键，确保 JOIN 所需关联字段不遗漏

• 输出结构化的 TableInfoState**表信息** 和 MetricInfoState**指标信息**，为后续 LLM 节点提供精准上下文

 

## 阶段四：LLM 精筛（并行）

 filter_table — LLM **去掉不需要的表和字段**  

- 将候选表/字段信息序列化为 YAML 传入 大模型，让模型判断哪些表、哪些字段真正需要

-  输出格式为 {表名: [字段名]} 的 JSON，过滤掉冗余字段

 filter_metric — **LLM 去掉不需要的指标**

- 通过大模型 从候选指标中选出本次问题必须使用的指标

 

## 阶段五：上下文增强（add_extra_context）

• 动态注入当前日期（年月日 + 星期 + 季度），支持「本月」「今年」等相对时间表达

• 从数据仓库查询数据库版本信息（dialect + version），确保生成 SQL 语法兼容性

 

## 阶段六：SQL 生成、校正、执行（generate → validate → correct/execute）

• generate_sql：将过滤后的表信息、指标定义、日期信息、DB 信息一起注入 Prompt提示词，调用大模型生成纯文本 SQL

• validate_sql：通过 EXPLAIN 语法预执行 SQL，检验语法与字段合法性；校验通过则路由到执行节点，校验失败则进入纠错节点

• correct_sql：将错误原因连同原始 SQL、上下文再次提交 给大模型，执行最小化修复，输出修正后 SQL

• execute_sql：异步执行 SQL，结果通过流写入器（stream_writer）实时将进度推送前端

 

# 四、技术架构亮点

## 亮点 1：多源RAG 检索策略

项目针对 NL2SQL 场景的三类不同语义对象，设计了三套并行召回通道，精准覆盖不同召回需求：

| 召回目标   | 检索引擎      | 检索方式            | 关键技术                           |
| ---------- | ------------- | ------------------- | ---------------------------------- |
| 字段语义   | Qdrant 向量库 | 余弦相似度向量检索  | Embedding + 余弦相似度（阈值过滤） |
| 指标概念   | Qdrant 向量库 | 余弦相似度向量检索  | 指标名/描述/别名联合向量化         |
| 字段枚举值 | Elasticsearch | IK 中文分词全文检索 | ik_max_word 分词 + min_score 过滤  |

LangGraph 并行边实现三路并行召回，最大程度缩短检索耗时，显著提升系统吞吐能力。

## 亮点 2：元数据驱动的知识库构建

系统维护两类存储：

• Meta MySQL：存储表/字段/指标的结构化元数据，包含字段类型、角色（主键/外键/度量/维度）、描述、别名、数据示例

• 索引层：通过 build_meta_knowledge 脚本将元数据批量向量化写入 Qdrant（字段信息、指标信息），字段值写入 Elasticsearch



元数据与索引层分离设计，元数据变更后只需重新构建知识库，无需改动业务代码，具备良好的可扩展性。

## 亮点 3：LangGraph 状态图编排 AI Agent

项目使用 LangGraph StateGraph 对整个 Agent 工作流进行声明式编排：

- 状态管理：DataAgentState 作为贯穿全图的共享上下文，各节点只读/写自身关注的字段，避免数据竞争
- 并行执行：extract_keywords → **(recall_column ‖ recall_value ‖ recall_metric**) → merge，天然支持并发
- 条件路由：**validate_sql** 节点根据执行结果动态路由至 execute_sql 或 correct_sql，实现 SQL 校正能力
- Context 注入：通过 DataAgentContext 将 DB 客户端、Repository 对象注入各节点，实现依赖解耦



## 亮点 4：Prompt 工程体系化设计

项目将所有 Prompt 外置为独立 .prompt 文件，分角色精细设计：

| Prompt 文件                       | 角色定位                               | 输出格式   |
| --------------------------------- | -------------------------------------- | ---------- |
| extend_keywords_for_column_recall | 字段推断专家：推断回答问题必需的字段名 | JSON 数组  |
| extend_keywords_for_metric_recall | 指标语义扩展专家：生成指标召回关键词   | JSON 数组  |
| extend_keywords_for_value_recall  | 业务语义解析专家：识别字段取值候选     | JSON 数组  |
| filter_table_info                 | 查询规划专家：裁剪必需的表和字段       | JSON 对象  |
| filter_metric_info                | 指标筛选专家：确定必需的指标           | JSON 数组  |
| generate_sql                      | 资深数据库专家：NL → SQL               | 纯文本 SQL |
| correct_sql                       | SQL 调试专家：最小化修复 SQL           | 纯文本 SQL |



每个 Prompt 均包含角色设定（Role)、任务说明(Action)、严格的生成规则(Format)和禁止事项，有效约束 LLM 输出格式，降低幻觉风险。



## 亮点 5：全异步高并发架构

• 框架层：FastAPI + Python asyncio，全异步 HTTP 服务

• 数据库层：SQLAlchemy AsyncSession（aiomysql 驱动），异步操作 Meta MySQL 和数据仓库 MySQL

• 向量库层：AsyncQdrantClient，异步向量检索

• ES 层：AsyncElasticsearch，异步全文检索

• LLM 层：LangChain ainvoke，异步调用大语言模型

所有 I/O 操作均为非阻塞，配合 LangGraph 并行节点，系统在高并发查询场景下具备良好的吞吐性能。

## 亮点 6：流式输出与进度实时反馈

• 每个 Agent 节点通过流写入器**stream_writer** 实时推送 {type: progress, step, status} 进度事件

• 最终查询结果以 {type: result, data} 事件流式写入前端，用户无需等待全流程完成

• LangGraph astream 接口支持 custom 流模式，前端基于SSE协议，可实时展示各阶段处理状态，提升用户体验

## 项目介绍

面试官：先介绍一下你做的这个项目吧

好的，这个项目叫 Data Agent，核心解决的问题就一句话：让不懂 SQL 的人也能查数据仓库。

用户用中文提问，比如「各省份今年的 GMV 是多少」，系统自动理解意图、从元数据里找到相关的表和字段、生成 SQL、执行查询，最后把结果返回来。整个过程用户完全感知不到 SQL 的存在。

技术上，它是一条基于 LangGraph 编排的多节点 AI Agent 流水线，里面集成了向量检索、全文检索、LLM 调用这几条路，最后通过 FastAPI 对外提供服务。

**说完之后可以主动问：「您想深入了解哪块？流程设计、还是某个具体的技术点？」**

面试官：这个项目是你一个人做的吗？做了多久？

是我独立完成的，从设计到实现大概花了2个月左右。

整体架构是我自己设计的，包括 Agent 流程怎么拆节点、三套检索通道怎么配合、Prompt 怎么写——这些都是边踩坑边调出来的。

面试官：用户输入一个问题，系统内部是怎么处理的？

整个流程我拆成了 12 个节点，用 LangGraph 串起来

**第一步**，关键字抽取。用 jieba 的  对问题分词，只保留名词、动词、专有名词这些有语义价值的词，然后把分词结果和原问题本身合并去重，得到初始关键字列表。

**第二步**，三路并行召回。这是整个系统里我觉得比较有意思的设计。三条路同时跑：

​	一路是字段语义召回，先让 LLM 把问题扩展成「字段名列表」，比如问「各省销售额」，LLM 会推断出「省份、销售金额、订单金额」这类字段名，然后把这些词向量化，去 Qdrant 向量库里做相似度检索，召回相关字段信息。

​	第二路是指标召回，同样让 LLM 扩展出指标概念词，比如「GMV、成交额、交易总额」，向量化后去 Qdrant 查指标知识库。

​	第三路是字段值召回，让 LLM 提取出问题里可能是字段取值的词，比如「华南、铂金会员、手机数码」，然后去 Elasticsearch 做中文全文检索。这条路解决的问题是：生成 SQL 的 WHERE 条件里，过滤值写「华南」还是「south」——得从真实数据里查。

**第三步**，信息融合。三路结果汇在一起，去重合并，还做了几件事：把 ES 召回的枚举值注入到对应字段的 examples 里，补全涉及表的主外键（否则 JOIN 会漏字段），从 Meta MySQL 里补全指标关联字段。

**第四步**，LLM 精筛。把融合后的候选表字段、候选指标都扔给 LLM，让它判断回答这个问题真正需要哪些，裁掉冗余的，同样并行跑。

**第五步**，上下文增强。动态注入当前日期（含星期和季度）和数据库版本——这俩是 LLM 生成带时间条件的 SQL 时必须知道的信息。

**第六步**，SQL 生成 → 验证 → 校正。LLM 生成 SQL，用 EXPLAIN 预执行校验语法，如果校验失败，把错误信息连带原 SQL 一起喂给另一个 Prompt 让 LLM 修复，修复完直接执行，结果流式推给前端。

## 技术细节

**面试官：**为什么用 LangGraph 而不是直接调 LLM？

需要多节点编排、条件路由、并行执行，LangGraph 的 StateGraph 比手写状态机更清晰可维护，且天然支持流式输出。

**面试官：**RAG 召回准确率怎么保证？

双路策略：Qdrant 向量召回保语义相似度，ES 保字段值精确匹配；加 LLM 关键词扩展增加召回入口；最终由 LLM 二次过滤，三层兜底。

**面试官：**SQL 生成错误怎么处理？

引入校验节点做语法检查，失败时将错误信息 + 原 SQL 拼回 Prompt 让 LLM 自我修复，形成闭环，避免直接报错中断。

**面试官：**Prompt 怎么防止 LLM 幻觉？

约束只用上下文中存在的表/字段；注入指标口径定义；限制输出格式（纯 SQL，不允许 Markdown）；角色设定为「资深数据库专家」提升精度。

**面试官：**为什么用 SSE 而不是普通 HTTP？

Agent 执行时间较长（多次 LLM 调用），SSE 允许服务端边执行边推送进度，用户体验远好于等待一次性返回，也比 WebSocket 轻量。

**面试官：**这个项目最难的地方是什么？

Schema 匹配问题：用户问法千变万化（「件数」「销量」「购买数量」都指同一字段），通过 LLM 扩展关键词 + 三路召回 + 字段别名体系三重保障解决。

### RAG 检索设计

**面试官：你为什么要用三套检索，一套向量检索不够吗？**

用一套肯定是最简单的，但我发现面对的问题其实是三类不同性质的语义，用同一套检索效果会很差。

字段名是「概念级」语义，比如「销售额」和「order_amount」，表达的是同一个意思，向量相似度能找到这种语义对齐，这块用 Qdrant 合适。

但字段的枚举取值，比如「华南」「铂金」，这是精确匹配的问题，不是语义近似的问题——「华南」和「华东」的向量相似度其实挺高的，用向量检索反而容易召回错。ES 的全文检索加 IK 分词反而更精准，而且性能更好。

指标这块是第三类，指标有很多别名，比如 GMV 也叫「成交额」「交易总额」，向量检索语义泛化能力强，这里用 Qdrant 也比较合理。

所以三套检索是针对三种不同语义特性专门设计的，不是凑数的。

**面试官：Qdrant 向量检索的相似度阈值 0.6 是怎么定的？**

说实话，0.6 是调出来的，不是拍脑袋定的。

阈值太低，会召回一堆不相关的字段，后续 LLM 精筛的压力就大了，而且容易带噪声进去；阈值太高，遇到用户问题措辞和字段描述差异比较大的情况，就会漏召。

我用了一批测试问题试了几个值，0.6 在我们这个数据集上召回率和精准率的平衡比较好。生产上如果数据变化大，这个值是需要根据数据重新校准的。

**面试官：ES 那边用了 IK 分词，为什么不用默认的 standard？**

standard 分词器是按字符切的，对中文来说基本就是把每个汉字单独切开，「华南」会变成「华」「南」两个词，查「华南大区」根本匹配不上。

IK 是中文专用分词器，能识别「华南」「铂金会员」「手机数码」这类业务词汇，匹配精度高很多。我用的是 **ik_max_word**，会对词做最细粒度切分，召回率更好。

 

###  LangGraph 与 Agent 设计

**面试官：为什么选 LangGraph，直接用 LangChain 的 Chain 不行吗？**

LangChain 的 Chain 是线性的，一步接一步，做不到「三路并行召回」这种结构。而且它对状态管理很弱，节点之间传数据很麻烦。

LangGraph 是基于状态图的，天生支持并行边，我只需要声明 extract_keywords 同时连到 recall_column、recall_value、recall_metric 三个节点，它就自动并行执行了，不需要我手动写 asyncio.gather。

另外 LangGraph 的条件路由很直观，validate_sql 根据 error 字段决定走 execute 还是走 correct，就是一个 lambda，逻辑很清晰。整体工作流声明式地写在 graph.py 里，一眼就能看出执行顺序。

**面试官：DataAgentState 和 DataAgentContext 有什么区别，为什么要分开？**

这个问题问得很好，我当时也纠结过。

State 是「流动的数据」——用户的问题、召回的字段、生成的 SQL、校验的错误，这些是每次请求都不一样的，需要在节点间传递和更新的。

Context 是「固定的基础设施」——数据库连接、向量库客户端、Repository 对象，这些是整个请求生命周期里共享的，不会变，本质上是依赖注入。如果把它们塞进 State，每次节点更新状态都要带着这些对象序列化，又重又乱。

分开之后，State 只存业务数据，Context 只存基础设施，各节点从 runtime.context 里取它需要的 Repository，职责很清晰。

**面试官：SQL 校正那里，为什么不做个循环多试几次？**

理论上可以循环，但我当时评估了一下：如果第一次生成的 SQL 加上错误信息让 LLM 修复还是不行，通常说明要么 Prompt 有根本性问题，要么上下文信息不够，再循环几次也大概率不行，反而白白多等几秒。

所以我目前的设计是「最多修复一次」，这在 validate → correct → execute 的边里体现了。如果要做多轮，LangGraph 支持带计数器的条件路由，加一个 retry_count 字段在 State 里就能实现，扩展很方便，但当前版本没开这个。

### Prompt 工程

**面试官：你的 Prompt 是怎么设计的？**

我把所有 Prompt 都外置成独立的 .prompt 文件，不硬编码在代码里，方便迭代。

每个 Prompt 基本都是三段式：角色定义、任务说明、输出格式约束。

角色定义是为了让 LLM 进入对应的「专家模式」——比如字段召回的 Prompt 里我写的是「你是一名数据表字段推断专家」，SQL 纠错的写的是「你是 SQL 调试专家，擅长根据错误信息最小化修复」，不同角色输出的质量差很多。

输出格式约束是重点。生成的 SQL 我要求「只输出纯文本 SQL，禁止 Markdown 代码块」，过滤类节点要求输出 JSON 数组或对象，然后接 JsonOutputParser 直接解析。格式不约束好，后处理就很麻烦。

另外每个 Prompt 里都有「禁止事项」，比如「禁止编造表或字段」「禁止输出解释性文本」，这能有效减少 LLM 幻觉。

**面试官：用 YAML 把上下文传给 LLM，为什么不用 JSON？**

主要考虑的是 token 效率和可读性。JSON 里有很多引号、花括号，YAML 对嵌套结构更紧凑，同样的信息 YAML 占的 token 少一些，LLM 理解起来也更清晰——毕竟 LLM 训练数据里 YAML 的结构化表达也很常见。

用了 allow_unicode=True 和 sort_keys=False，保证中文字段描述不被转义、字段顺序保持原始定义顺序，这两个细节蛮重要的。



### 异步架构

**面试官：你说全链路异步，具体体现在哪？**

从上到下走一遍：

接入层是 FastAPI，原生支持 async/await，接口定义成 async def；

Agent 层，LangGraph 的 astream 方法是异步的，每个节点函数都是 async def；

LLM 调用用的 chain.ainvoke，LangChain 的异步调用接口；

数据库用的 SQLAlchemy AsyncSession 加 aiomysql 驱动，execute 方法加 await；

向量库用的 AsyncQdrantClient，ES 用的 AsyncElasticsearch，都是原生异步客户端。

所以整条链路没有一个阻塞的 I/O——用户请求进来，线程不会卡在等数据库或等 LLM 返回上，可以继续处理其他请求。

**async/await 与同步代码的核心区别？**

同步代码的执行逻辑是**“串行的、阻塞的”**：

- 代码从上到下一行行执行，每一行必须等上一行执行完毕才能开始。
- 如果遇到 “耗时操作”（比如网络请求、文件读写、数据库查询），整个程序会停下来等待这个操作完成，期间 CPU 处于闲置状态。

async/await 是**异步编程的语法糖**，基于 “协程（Coroutine）” 实现，核心逻辑是**“非阻塞的、并发的”**：

- 代码遇到 “耗时操作” 时，不会阻塞整个程序，而是把 CPU 让给其他任务去执行。
- 当耗时操作完成后，再回来继续执行剩下的代码。
- `async` 标记函数为 “协程函数”（执行后返回协程对象），`await` 用于 “等待异步操作完成”（但不会阻塞事件循环）。
- 把等待的时间用在处理其他任务上

**面试官：流式输出是怎么实现的？**

LangGraph 的 astream 支持自定义流模式，我用的是 stream_mode="custom"。

每个节点拿到 runtime.stream_writer 这个流写入器，处理开始时推一个 {type: progress, step: 节点名, status: running}，处理完推 success，出错推 error。执行完 SQL 之后推 {type: result, data: 查询结果}。

这些事件会实时流出来，基于SSE协议：所有数据格式"data: 数据\n\n"前端订阅之后可以做进度条——用户看到「正在召回字段…」「正在生成 SQL…」这样的实时反馈，体验比等一个转圈圈等待突然返回结果好很多。

 

### 项目经验参考

项目名称：基于 LLM + RAG 的企业级自然语言数据查询 Agent（NL2SQL）

项目背景：面向业务人员的数据自助分析平台，解决传统 BI 工具依赖 SQL 专业知识的门槛问题，实现【问一句话，出一张表】的零门槛查询体验。

技术栈：Python、LangGraph、LangChain、Qdrant、Elasticsearch、FastAPI、HuggingFace Embedding、SQLAlchemy、Jieba、SSE 流式推送、OmegaConf 

核心职责与技术亮点：

• 【Agent 架构设计】基于 LangGraph StateGraph 设计 12 个节点有向图 Agent，节点间通过共享状态传递上下文，支持条件路由（SQL 校验失败自动触发纠错节点），实现端到端的查询 Pipeline 编排。

• 【RAG 增强召回】构建元数据知识库，使用 HuggingFace Embedding 模型对字段信息、业务指标进行向量化，存储至 Qdrant；字段值存储至 Elasticsearch 支持模糊检索；通过 LLM 驱动的关键词扩展提升召回覆盖率。

• 【业务语义注入】设计表信息、字段信息、指标信息三层元数据体系，将 GMV、AOV 等业务指标的口径和计算逻辑结构化存储，动态注入 Prompt，减少 SQL 生成幻觉。

• 【SQL 生成与纠错】使用 LLM（Deepseek）结合精心设计的 Prompt 模板生成 SQL，引入 SQL 语法校验节点，异常时触发 LLM 自动纠错，保障 SQL 可执行性。

• 【流式接口设计】基于 FastAPI + SSE 实现流式接口，Agent 执行各阶段实时推送进度事件（抽取关键字/召回字段/生成SQL 等），提升前端交互体验。

【工程化实践】采用 Repository 模式解耦数据访问层，支持 MySQL / Qdrant / ES 多数据源；使用 SQLAlchemy 异步 ORM；通过 OmegaConf 统一管理配置。

# 六、难点 & 踩坑&遇到的问题

面试官：做这个项目遇到什么比较难的问题？

一、印象最深的是「字段取值」这块踩的坑。早期版本没有 ES 这条召回路，LLM 生成 SQL 的时候不知道数据库里实际存的是什么值，比如DB中地区字段可能存的是「华北」，用户说的是「华北地区」，LLM 就可能写成 WHERE region_name = '华北地区'，SQL 能跑但查不出数据，程序不会发生异常，很难发现。

后来加了 ES 全文检索这条路，把真实的字段取值索引进去，召回之后注入到对应字段的 examples 里，LLM 看到 examples 里有「华北」就会用正确的值，这个问题才真正解决。



二、另一个坑是 JOIN 漏主外键。比如我这个项目，发现大模型对召回的表错误的关联。早期 merge合并召回信息 节点没有「自动补全主外键」的逻辑，有时候召回的字段里没有 region_id 这种外键，LLM 就不知道怎么 JOIN，生成的 SQL 要么用字段名直接匹配（全表扫），要么乱 JOIN。后来加了「自动从 Meta DB 查涉及表的主外键并补全」的逻辑才解决。

**这两个坑都能从代码里证明——ES 召回 + examples 注入，以及 merge 节点里的 get_key_columns_by_table_id 调用获取主外键字段**
