Chapter 1

ElasticSearch 简介

1.1 什么是 ElasticSearch

Elasticsearch，简称为 ES，是一个开源的高扩展分布式全文检索引擎。它可以近乎实时的存储、检索数据；本身扩展性很好，可以扩展到上百台服务器，处理 PB 级别的数据。

ES 核心特点
  • 使用 Java 开发，基于 Lucene 作为核心索引与搜索引擎
  • 通过简单的 RESTful API 隐藏 Lucene 的复杂性
  • 分布式架构，可横向扩展到上百台服务器
  • 近实时（Near Real-Time，NRT）搜索
  • 处理 PB 级别的海量数据

1.1.1 什么是全文检索

数据分类

📊 结构化数据

具有固定格式或有限长度的数据

  • 数据库表（MySQL/Oracle）
  • 元数据
  • CSV/Excel 表格

✓ 搜索简单：用 SQL 即可快速查询

📝 非结构化数据

不定长或无固定格式的数据

  • 邮件、Word/PDF 文档
  • 网页内容、日志文件
  • 社交媒体文本

✗ 搜索困难：需要全文检索技术

两种查询方法对比

🐢 顺序扫描法（Sequential Scan）

一个一个文档地看，从头看到尾，判断是否包含目标字符串。

❌ 数据量大时极慢，如 Windows 文件内容搜索

🚀 全文检索（Full-Text Search）

先对非结构化数据提取信息 → 建立索引，再对索引进行搜索。

✓ 索引创建耗时长，但创建后可多次使用，搜索极快

💡 形象理解：全文检索就像查字典——字典的拼音检字表就是"索引"，通过索引快速定位页码，而不是一页页翻。先建立索引，再对索引搜索的过程就叫全文检索。

1.1.3 倒排索引 — ES 的核心秘密

传统方法是"根据文件找内容"（顺序扫描），而倒排索引是"根据内容（词语）找文件"，这是 ES 搜索飞快的根本原因。

词频(TF)等信息 → 用于搜索结果的相关性排序打分 📊 倒排索引数据结构（实际存储） 单词ID | 单词 | 倒排列表(文档ID, 词频TF, 位置...) 1 | 谷歌 | (1,1,<1>), (2,1,<1>), (3,1,<1>) 2 | 地图 | (2,1,<3>), (3,1,<3>)
📌 关键理解：倒排索引 = 索引(词汇表，规模小) + 文档(文档集合，规模大)。索引系统通过记录词频(TF)等信息，可以在搜索结果排序时计算查询与文档的相似度得分，得分越高排在越前面。

1.1.4 ES vs MySQL 核心对比

🐬 MySQL（关系型数据库）
  • LIKE 模糊查询遍历全表，数据量大时极慢
  • 不支持分词，中文组合词搜索困难
  • 不支持相关性排序
  • 适合结构化数据的精确查询

定位：负责存储数据

🔍 ElasticSearch（搜索引擎）
  • 基于倒排索引，TB级数据毫秒级返回
  • 内置中文分词（IK分词器），智能拆词
  • 相关性评分，最相关的结果排前面
  • 分布式实时搜索，水平扩展

定位：负责搜索数据

经典案例：MySQL 与 ES 分工
场景 MySQL 表现 ES 表现
搜索"手机" 遍历所有记录逐一匹配 倒排索引直接定位
搜索"四川火锅" 只能全词匹配，无法拆词 自动分词"四川"+"火锅"
搜索"星巴克咖啡" 无法按相关性排序 相关性评分，精确匹配排前

1.2 ES 名企使用案例

  • GitHub：2013年抛弃 Solr 转向 ES，搜索 20TB 数据，含 13亿文件 和 1300亿行代码
  • 维基百科：以 ES 为核心搜索架构
  • SoundCloud：为 1.8亿用户 提供实时精准音乐搜索
  • 百度：覆盖 20+ 业务线，最大集群 100台机器 / 200个ES节点，每天导入 30TB+ 数据
  • 新浪：分析处理 32亿条实时日志
  • 阿里：构建日志采集和分析体系

ElasticSearch 安装与启动

2.1 Windows 版本安装

  1. 下载 ES 压缩包

    官方地址：https://www.elastic.co/products/elasticsearch，课程使用 7.4.0 版本

  2. 解压即安装

    Window 版解压即完成安装，目录结构包含 bin、config、lib、modules、plugins 等

    在 plugins 目录下创建 ik 文件夹，将 elasticsearch-analysis-ik-7.4.0.zip 解压进去

  4. 启动 ES 服务

    双击 bin/elasticsearch.bat，控制台输出日志

    ⚠️ 前置条件：需要 JDK 1.8+，并正确配置 JAVA_HOME 环境变量

2.4 IK 分词器详解

IKAnalyzer 是基于 Java 的轻量级中文分词工具包，60万字/秒处理速度。

ik_max_word（最细粒度）

将文本做最细粒度拆分，尽可能多地分出词汇。

GET /_analyze
{
  "analyzer": "ik_max_word",
  "text": "乒乓球明年总冠军"
}

结果：乒乓球 | 乒乓 | 球 | 明年 | 总冠军 | 冠军

ik_smart（粗粒度）

做粗粒度拆分，更智能简洁。

GET /_analyze
{
  "analyzer": "ik_smart",
  "text": "乒乓球明年总冠军"
}

结果：乒乓球 | 明年 | 总冠军

💡 选型建议：ik_max_word 召回率更高（搜到更多结果），ik_smart 精确度更高（搜到更准的结果）。实际开发中根据业务场景选择。
Chapter 3

ElasticSearch 核心概念（术语）

3.1 ES 与 MySQL 概念类比

ES 是面向文档型数据库，一条数据就是一个 JSON 文档。

Type（类型）← 6.x已废弃！ Document（文档）← 类似 Row Field（字段）← 类似 Column

3.2 五大核心概念

① Index（索引）

文档存储的地方，类似于 MySQL 中的 Database 概念。一个 Elasticsearch 集群可以包含多个索引。

② Mapping（映射）

对文档中每个字段的类型进行定义，类似于 MySQL 中的表结构定义（Schema）。

ES 实现了动态映射（Dynamic Mapping），大多数情况下不需要手动创建映射，ES 会自动推断字段类型。

📌 关键类型：text（需分词，如文章内容）vs keyword（不分词，如邮箱、标签）。ES 5.x 起废弃了 string 类型。
③ Document（文档）

ES 中的最小数据单元，类似于 MySQL 中的一行数据，以 JSON 格式表示。每个文档都有唯一 ID 和元数据（_index, _id, _version 等）。

{
  "name": "jack",
  "age": 18,
  "gender": 1
}
④ Field（字段）

相当于数据表的列，JSON 数据中的 Key。不同类型的字段有不同的使用场景。

⑤ Type（类型）— 已废弃

重要 ES 6.0 之前有 Type（表）的概念，但官方认为这是设计失误，6.0 之后已废弃。建立索引后直接往索引中写入文档即可。

Chapter 4

客户端操作 — RESTful API

ES 提供 RESTful 风格的 API，所有操作都通过 HTTP 请求完成。主要有三种客户端方式：elasticsearch-head 插件、Postman / Kibana 直接发请求、Java API。

4.1 索引（Index）CRUD 操作

PUT  localhost:9200/blog           // 创建索引
GET  localhost:9200/blog           // 查看索引
DEL localhost:9200/blog           // 删除索引
POST localhost:9200/blog/_close    // 关闭索引
POST localhost:9200/blog/_open     // 打开索引

4.2 创建索引并设置 Mapping

PUT localhost:9200/blog1
{
    "mappings": {
        "properties": {
            "id":      { "type": "long", "store": true, "index": true },
            "title":   { "type": "text", "store": true, "index": true, "analyzer": "standard" },
            "content": { "type": "text", "store": true, "index": true, "analyzer": "standard" }
        }
    }
}

4.3 Mapping 单独操作

GET  person/_mapping                 // 查询映射
PUT  person/_mapping                 // 添加映射
{
  "properties": {
    "name": { "type": "keyword" },
    "age":  { "type": "integer" }
  }
}
PUT  person/_mapping                 // 索引库中添加字段
{
  "properties": { "address": { "type": "text" } }
}

4.4 文档（Document）CRUD

PUT  person/_doc/1                   // 添加文档(指定id)，id存在则更新
{ "name":"张三", "age":20, "address":"深圳宝安区" }

POST person/_doc/                    // 添加文档(不指定id)，自动生成id
{ "name":"李四", "age":20, "address":"深圳南山区" }

GET  person/_doc/1                   // 根据id查询文档
GET  person/_search                  // 查询所有文档
DEL person/_doc/1                   // 删除文档

4.5 核心查询方式对比

🔍 Match 查询（全文查询）

会分析查询条件，先将查询字符串分词，再查询，求并集。

GET person/_search
{
  "query": {
    "match": {
      "address": "深圳保税区"
    }
  }
}

"深圳保税区" → 分词为 "深圳"、"保税区" → 包含任意一个词的文档都返回

🎯 Term 查询（词条查询）

不会分析查询条件，只有完全匹配词条时才返回。

GET person/_search
{
  "query": {
    "term": {
      "address": { "value": "深圳南山区" }
    }
  }
}

强调：查的是索引中的"词条"而非原文，与分词器有关

4.6 Bool 查询（组合查询）

GET shangguigu/_search
{
  "query": {
    "bool": {
      "filter": {                    // 过滤条件（不参与评分）
        "range": { "age": { "gt": 20 } }
      },
      "must": {                      // 必须匹配（参与评分）
        "match": { "sex": "女" }
      }
    }
  }
}

Bool 查询四种子句：

子句说明
must必须匹配，贡献评分
filter必须匹配，不贡献评分（性能更好）
should应该匹配（OR 逻辑），贡献评分
must_not必须不匹配，不贡献评分

4.7 其他常用查询

Range 范围查询

支持 gt(大于)、gte(大于等于)、lt(小于)、lte(小于等于)

POST atguigu/_search
{ "query": { "range": { "age": { "gte": 20, "lte": 22 } } } }
Terms 多词条查询
POST atguigu/_search
{ "query": { "terms": { "age": [20, 21] } } }
Exists 存在性查询
POST atguigu/_search
{ "query": { "exists": { "field": "mail" } } }

4.8 高级功能

高亮显示（Highlight）
GET shangguigu/_search
{
  "query": { "match": { "name": "张三 李四" } },
  "highlight": { "fields": { "name": {} } }
}
聚合查询（Aggregation）— 类似 SQL GROUP BY
GET shangguigu/_search
{
  "aggs": {
    "all_ages": {
      "terms": { "field": "age.keyword" }
    }
  }
}
⚠️ 注意：对 text 类型字段聚合需要加 .keyword 后缀，或开启 fielddata（不推荐，消耗堆内存）。
分页（Pagination）

size：每页结果数（默认10）；from：跳过的结果数（默认0）

GET /_search?size=5             // 第1页
GET /_search?size=5&from=5       // 第2页
GET /_search?size=5&from=10      // 第3页
指定响应字段与文档存在判断
GET  shangguigu/_doc/1001?_source=id,name     // 只返回指定字段
HEAD shangguigu/_doc/1001                     // 判断文档是否存在(200/404)
批量操作（_bulk）

批量插入、修改、删除，减少网络请求。格式：action行 + JSON数据行交替。

POST _bulk
{"create":{"_index":"atguigu","_id":2001}}
{"id":2001,"name":"name1","age":20,"sex":"男"}
{"create":{"_index":"atguigu","_id":2002}}
{"id":2002,"name":"name2","age":20,"sex":"男"}

POST _bulk    // 批量删除（delete 没有请求体）
{"delete":{"_index":"atguigu","_id":2001}}
{"delete":{"_index":"atguigu","_id":2002}}
📌 index vs create：index 对已有文档会覆盖（_version递增），create 对已有文档会失败抛出异常。

4.9 关键字搜索（URL参数方式）

GET  person/_search?q=name:张三
Chapter 5

ElasticSearch 集群

5.1 为什么需要集群？

单节点 ES 存在三大问题：存储容量有限、单点故障无高可用、并发处理能力有限。集群中节点数量一般 3 个以上。

5.2 集群核心概念

Cluster（集群）

由一个或多个节点组成，共同持有数据并提供索引和搜索功能。集群由唯一名字标识，默认名为 "elasticsearch"。

Node（节点）

集群中的单台服务器，存储数据、参与索引和搜索。节点名默认是随机漫威角色名。通过配置集群名称加入指定集群。

Shards & Replicas（分片与副本）

索引可划分为多个分片（Shard）分散存储在不同节点上，每个分片是一个功能完善的"小索引"。

每个分片可以有多个副本（Replica），副本从不与原分片放在同一节点。

分片两大作用：① 水平扩展容量；② 分布式并行操作提高吞吐量
副本两大作用：① 故障转移高可用；② 搜索可在副本上并行执行
⚠️ 限制：分片数量创建后不可更改（需重建索引），副本数量可以动态修改。
默认：ES 7.x 每个索引 1 个主分片 + 1 个副本 = 2 个分片（ES 6.x 为 5+1=10）

5.3 集群健康状态

5.4 集群搭建步骤（3节点示例）

  1. 复制三份 ES到 elasticsearch-cluster 文件夹下（node1/node2/node3）
  2. 修改各节点配置 elasticsearch.yml：
// node1 配置
cluster.name: my-es
node.name: node-1
network.host: 127.0.0.1
http.port: 9201
transport.port: 9301
discovery.seed_hosts: ["127.0.0.1:9302","127.0.0.1:9303"]
cluster.initial_master_nodes: ["node-1"]

// node2 / node3 类似，修改端口和节点名
  1. 清理旧数据：删除各 node 的 data 目录
  2. 安装 IK 分词器到各节点
  3. 依次启动各节点的 elasticsearch.bat

6.1 项目环境搭建

使用 Spring Boot 2.2.2 + Elasticsearch 7.4.0 RestHighLevelClient。

Maven 依赖

<dependency>
    <groupId>org.elasticsearch.client</groupId>

<dependency>

<dependency>
    <groupId>org.elasticsearch</groupId>

配置类 — ElasticSearchConfig

@Configuration
@ConfigurationProperties(prefix = "elasticsearch")
public class ElasticSearchConfig {
    private String host;    // 127.0.0.1
    private int port;       // 9200

    @Bean
    public RestHighLevelClient client() {
        return new RestHighLevelClient(
            RestClient.builder(new HttpHost(host, port, "http"))
        );
    }
    // getters/setters...
}

6.2 索引操作（Java API）

// 1. 创建索引
IndicesClient indicesClient = client.indices();
CreateIndexRequest request = new CreateIndexRequest("abc");
CreateIndexResponse response = indicesClient.create(request, RequestOptions.DEFAULT);

// 2. 创建索引 + 设置 Mapping
CreateIndexRequest request = new CreateIndexRequest("aaa");
String mapping = "{\"properties\": { \"name\": {\"type\": \"keyword\"}, ... }}";
request.mapping(mapping, XContentType.JSON);
indicesClient.create(request, RequestOptions.DEFAULT);

// 3. 查询索引
GetIndexRequest request = new GetIndexRequest("aaa");
GetIndexResponse response = indicesClient.get(request, RequestOptions.DEFAULT);

// 4. 判断索引是否存在
indicesClient.exists(new GetIndexRequest("aaa"), RequestOptions.DEFAULT);

// 5. 删除索引
indicesClient.delete(new DeleteIndexRequest("abc"), RequestOptions.DEFAULT);

6.3 文档操作（Java API）

// 1. 添加文档
IndexRequest request = new IndexRequest("person").id("1");
request.source("{\"name\":\"张三\",\"age\":20}", XContentType.JSON);
IndexResponse response = client.index(request, RequestOptions.DEFAULT);

// 2. 查询文档
GetRequest request = new GetRequest("person", "1");
GetResponse response = client.get(request, RequestOptions.DEFAULT);

// 3. 更新文档（id存在即更新）
UpdateRequest request = new UpdateRequest("person", "1");
request.doc("{\"age\":21}", XContentType.JSON);
client.update(request, RequestOptions.DEFAULT);

// 4. 删除文档
DeleteRequest request = new DeleteRequest("person", "1");
client.delete(request, RequestOptions.DEFAULT);

// 5. 批量操作
BulkRequest bulkRequest = new BulkRequest();
bulkRequest.add(new IndexRequest("person").id("1").source(...));
bulkRequest.add(new IndexRequest("person").id("2").source(...));
client.bulk(bulkRequest, RequestOptions.DEFAULT);

6.4 查询操作（Java API）

// 1. 查询所有
SearchRequest request = new SearchRequest("person");
SearchSourceBuilder builder = new SearchSourceBuilder();
builder.query(QueryBuilders.matchAllQuery());
request.source(builder);
SearchResponse response = client.search(request, RequestOptions.DEFAULT);

// 2. Match 查询
builder.query(QueryBuilders.matchQuery("address", "深圳"));

// 3. Term 查询
builder.query(QueryBuilders.termQuery("name", "张三"));

// 4. Bool 组合查询
BoolQueryBuilder boolQuery = QueryBuilders.boolQuery();
boolQuery.must(QueryBuilders.matchQuery("sex", "女"));
boolQuery.filter(QueryBuilders.rangeQuery("age").gt(20));

// 5. 范围查询
builder.query(QueryBuilders.rangeQuery("age").gte(20).lte(30));

// 6. 分页 + 排序
builder.from(0).size(10);
builder.sort("age", SortOrder.DESC);

// 7. 高亮
HighlightBuilder highlightBuilder = new HighlightBuilder();
highlightBuilder.field("name");
builder.highlighter(highlightBuilder);

// 8. 聚合
builder.aggregation(AggregationBuilders.terms("age_group").field("age"));
Chapter 7

Spring Data Elasticsearch

7.1 简介

Spring Data Elasticsearch 基于 Spring Data API 简化 ES 操作，将原始客户端 API 进行封装。类似于 Spring Data JPA 对数据库操作的简化——定义接口即可完成 CRUD，几乎不用写实现。

📌 重要变化：新版本中 ElasticsearchRestTemplate 替代了 ElasticsearchTemplate（后者基于即将废弃的 TransportClient）。配置类需继承 AbstractElasticsearchConfiguration。

7.2 Maven 依赖

<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-data-elasticsearch</artifactId>
</dependency>

7.3 实体类注解

@Document(indexName = "item", shards = 1, replicas = 1)
public class Item {
    @Id
    private Long id;

    @Field(type = FieldType.Text, analyzer = "ik_max_word")
    private String title;         // 标题 — 需分词

    @Field(type = FieldType.Keyword)
    private String category;      // 分类 — 不分词

    @Field(type = FieldType.Keyword)
    private String brand;         // 品牌 — 不分词

    @Field(type = FieldType.Double)
    private Double price;         // 价格

    @Field(index = false, type = FieldType.Keyword)
    private String images;        // 图片地址 — 不索引
}
三大注解说明
注解作用域属性
@Document 类 indexName(索引名), shards(分片数), replicas(副本数)
@Id 字段 标记为主键
@Field 字段 type(字段类型), index(是否索引), store(是否存储), analyzer(分词器)

7.4 Repository 接口 — CRUD 零代码

只需要定义一个接口继承 ElasticsearchRepository，即可自动拥有全部 CRUD 方法：

public interface ItemRepository extends ElasticsearchRepository<Item, Long> {
    // 继承的方法自动实现：save(), saveAll(), findById(), findAll(), deleteById(), count()...
}
Repository 自带方法
// 新增/修改（id存在则修改，不存在则新增）
itemRepository.save(item);

// 批量新增
itemRepository.saveAll(list);

// 根据id查询
Optional<Item> optional = itemRepository.findById(2L);

// 查询全部
Iterable<Item> all = itemRepository.findAll();

// 根据id删除
itemRepository.deleteById(1L);

@Autowired

// 创建索引 + 自动映射（根据 @Document 和 @Field 注解）
elasticsearchTemplate.createIndex(Item.class);
elasticsearchTemplate.putMapping(Item.class);

7.6 Spring Data Elasticsearch vs RestHighLevelClient

Spring Data Elasticsearch
  • 注解驱动，声明式开发
  • 继承 Repository 即可 CRUD
  • 与 Spring 生态无缝集成
  • 复杂查询灵活度有限
RestHighLevelClient
  • 完全控制查询细节
  • 支持所有 ES API
  • 适合复杂搜索场景
  • 代码量较大
