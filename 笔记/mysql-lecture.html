一、MySQL 简介与安装

📖 MySQL 概述

MySQL 是一个关系型数据库管理系统，由瑞典 MySQL AB（创始人 Michael Widenius）开发，2008 年被 Sun 收购（10 亿美金），2009 年 Sun 被 Oracle 收购。

MySQL 是开源的，采用 GPL 协议，支持处理上千万条记录的大型数据库——32 位系统表文件最大 4GB，64 位系统最大支持 8TB。

衍生版本 MariaDB 基于事务的 Maria 存储引擎替换了 MyISAM，使用 Percona 的 XtraDB 代替了 InnoDB。阿里巴巴大部分 MySQL 数据库实际使用 percona 的原型加以修改（AliSQL + AliRedis）。

💿 CentOS7 安装 MySQL 5.7.28 (RPM)
# 1. 检查并卸载旧版本
rpm -qa | grep -i mariadb
rpm -e --nodeps mariadb-libs

# 2. 检查依赖
rpm -qa|grep libaio
rpm -qa|grep net-tools

# 3. 按顺序安装（/opt 目录下）
rpm -ivh mysql-community-common-5.7.28-1.el7.x86_64.rpm
rpm -ivh mysql-community-libs-5.7.28-1.el7.x86_64.rpm
rpm -ivh mysql-community-client-5.7.28-1.el7.x86_64.rpm
rpm -ivh mysql-community-server-5.7.28-1.el7.x86_64.rpm

# 4. 初始化（生成临时 root 密码）
mysqld --initialize --user=mysql
cat /var/log/mysqld.log   # 查看临时密码

# 5. 启动并设置自启
systemctl start mysqld.service
systemctl enable mysqld.service

# 6. 首次登录并修改密码
mysql -uroot -p
ALTER USER 'root'@'localhost' IDENTIFIED BY 'new_password';
🔤 字符集配置
# 编辑 /etc/my.cnf，[mysqld] 下添加：
character_set_server=utf8
init_connect='SET NAMES utf8'

# 重启生效
systemctl restart mysqld

# 修改已有库表字符集
ALTER DATABASE mydb CHARACTER SET 'utf8';
ALTER TABLE mytbl CONVERT TO CHARACTER SET 'utf8';

# 查看字符集
SHOW VARIABLES LIKE 'character%';

二、用户与权限管理

👤 用户管理
-- 创建用户
CREATE USER zhang3 IDENTIFIED BY '123123';

-- 查看用户
SELECT host,user,authentication_string FROM mysql.user;

-- 修改密码（当前用户）
SET PASSWORD = PASSWORD('123456');

-- 修改密码（root 修改其他用户）
UPDATE mysql.user SET authentication_string=PASSWORD('111111')
WHERE user='zhang3';
FLUSH PRIVILEGES;   -- 必须执行，否则不生效

-- 删除用户（推荐）
DROP USER zhang3;
-- 不要用 DELETE FROM mysql.user，会有残留信息
🔐 权限管理
-- 授权（如用户不存在则自动创建）
GRANT 权限1,权限2,... ON 数据库.表 TO 用户@地址 IDENTIFIED BY '密码';

-- 例：给 li4 本地用户授予所有库表的 CRUD 权限
GRANT SELECT,INSERT,UPDATE,DELETE ON *.* TO li4@localhost IDENTIFIED BY '123';

-- 例：给 joe 远程用户授予全部权限（含授权能力）
GRANT ALL PRIVILEGES ON *.* TO joe@'%' IDENTIFIED BY '123'
WITH GRANT OPTION;

-- 收回权限
REVOKE ALL PRIVILEGES ON *.* FROM joe@'%';
REVOKE SELECT,INSERT,UPDATE,DELETE ON *.* FROM joe@'%';

-- 查看权限
SHOW GRANTS;
SELECT * FROM mysql.user WHERE user='root' AND host='%';
host 值含义
%所有远程 TCP 方式连接
192.168.1.2指定 IP 的 TCP 连接
localhost本地命令行方式连接
::1IPv6 本地地址（等同 127.0.0.1）
⚠️ 注意：MySQL 5.7 密码保存在 authentication_string 字段中，不再使用 password 字段。所有通过 UPDATE mysql.user 的修改必须执行 FLUSH PRIVILEGES 才能生效。

💡 关键点：查询流程为：客户端连接 → 检查查询缓存（命中直接返回）→ 解析器生成解析树 → 预处理器检查合法性 → 优化器生成执行计划 → 存储引擎执行 → 返回结果。
注意：INSERT/UPDATE/DELETE 会使缓存失效；SQL 必须完全一致才能命中缓存。

四、索引数据结构：从二叉树到 B+Tree

🌳 数据结构演进路线
📐 B-Tree vs B+Tree 对比
特性B-TreeB+Tree
数据存储位置所有节点都存数据只有叶子节点存数据
非叶子节点内容键值 + 数据仅键值 + 指针
叶子节点连接无连接双向链表相连
范围查询效率需多次回溯叶子链顺序遍历，极快
磁盘块利用率较低（存数据占空间）更高（非叶子纯指针）
树高度较高更矮胖
使用场景文件系统MySQL InnoDB、操作系统磁盘
💡 为什么索引选树不选 Hash？Hash 的单点查询 O(1) 比树快，但遇到 ORDER BY、BETWEEN、>、< 等范围/排序查询时退化为 O(n)；而 B+Tree 因为有序列特性，始终能保持 O(log₂N) 的高效率。InnoDB 不支持 Hash 索引也是这个原因。

六、索引优化分析

📇 索引分类与语法
-- 单值索引
CREATE INDEX idx_name ON emp(name);

-- 唯一索引（值唯一，允许NULL）
CREATE UNIQUE INDEX idx_email ON emp(email);

-- 主键索引（建表自动创建，InnoDB为聚簇索引）
ALTER TABLE emp ADD PRIMARY KEY (id);

-- 复合索引（多列）
CREATE INDEX idx_age_name ON emp(age, name);

-- 删除索引
DROP INDEX idx_name ON emp;

-- 查看索引
SHOW INDEX FROM emp;

-- ALTER 方式添加
ALTER TABLE emp ADD INDEX idx_name (name);
ALTER TABLE emp ADD UNIQUE idx_email (email);
ALTER TABLE emp ADD FULLTEXT idx_content (content);
📊 聚簇索引 vs 非聚簇索引
特性聚簇索引非聚簇索引
数据存储数据行和索引存储在一起索引和数据分开存储
叶子节点存完整行数据存主键值（再去聚簇索引查）
每表数量只能1个（一般是主键）可以有多个
支持引擎InnoDBMyISAM
主键建议用有序自增ID—
优点范围查询快（数据连续）—
🔴 索引失效十大场景（重磅！）

✅ 1. 全值匹配我最爱 — 复合索引所有字段都用上，效率最高

EXPLAIN SELECT * FROM emp WHERE age=30 AND deptid=4 AND name='abcd';
-- 使用 idx_age_deptid_name 全部三个字段 ✓

❌ 2. 最佳左前缀法则 — 复合索引必须从最左列开始，且不能跳过中间列

EXPLAIN SELECT * FROM emp WHERE age=30 AND name='abcd';
-- 跳过了 deptid，name 索引失效 ✗（只能用 age）

❌ 3. 计算/函数导致索引失效

SELECT * FROM emp WHERE LEFT(name,3)='abc';  -- ✗ 索引失效
SELECT * FROM emp WHERE name LIKE 'abc%';      -- ✓ 索引可用

❌ 4. 范围条件右边的列索引失效

-- idx_age_deptid_name：deptid 用了范围，name 失效
SELECT * FROM emp WHERE age=30 AND deptid>20 AND name='abc';
-- 优化：把范围查询放最后 → CREATE INDEX idx_age_name_deptid

❌ 5. != 或 <> 索引失效

❌ 6. IS NOT NULL 索引失效（IS NULL 可用）

❌ 7. LIKE 以 % 开头索引失效

SELECT * FROM emp WHERE name LIKE 'ab%';   -- ✓ 可用
SELECT * FROM emp WHERE name LIKE '%ab%';  -- ✗ 失效

❌ 8. 类型转换导致索引失效

SELECT * FROM emp WHERE name=123;    -- ✗ 字符串列用数字比较
SELECT * FROM emp WHERE name='123';   -- ✓

❌ 9. OR 连接 — 如果 OR 两边有一个没有索引，整个查询不使用索引

✅ 10. 覆盖索引 — 查询列被索引覆盖时，即使有部分条件不符合索引规范，也可能用到索引

🔥 面试重点 — 索引失效口诀：
全值匹配我最爱，最左前缀要遵守；
带头大哥不能死，中间兄弟不能断；
索引列上少计算，范围之后全失效；
LIKE百分写最右，覆盖索引不写星；
不等空值还有OR，索引失效要注意；
VAR引号不可丢，SQL高级也不难！

九、排序与分组优化

🎯 ORDER BY 优化三法则
-- 法则1：无过滤，不索引
SELECT * FROM emp ORDER BY age,name;  -- ✗ 无WHERE，不走索引

-- 法则2：顺序错，必排序
-- 索引 idx_age_name，下面两个SQL对比：
SELECT * FROM emp WHERE age=30 ORDER BY name;   -- ✓ 符合索引顺序
SELECT * FROM emp WHERE age=30 ORDER BY name,age;  -- ✗ 顺序反，filesort

-- 法则3：方向反，必排序
SELECT * FROM emp WHERE age=30 ORDER BY age ASC,name DESC;  -- ✗ 方向不一致
🥊 范围 vs 排序 — 经典二选一

当 WHERE 中有范围条件 + ORDER BY 时，索引只能二选一（范围右边的列索引会失效）。

决策原则：优先观察条件字段的过滤数量。如果范围条件过滤掉大部分数据（如只剩几百条），排序性能消耗不大，应把索引放在范围字段上。反之如果过滤后仍有很多数据，索引应放在排序字段上。
-- 场景：age=30 AND empno<101000 ORDER BY name
-- age 过滤后剩大量数据 → 应建 idx_age_name（放弃范围索引）
CREATE INDEX idx_age_name ON emp(age,name); -- 推荐

-- 如果 empno 过滤掉大部分数据 → 应建 idx_age_empno（放弃排序索引）
CREATE INDEX idx_age_empno ON emp(age,empno); -- 视情况
📚 双路排序 vs 单路排序
特性双路排序 (MySQL 4.1前)单路排序 (MySQL 4.1+)
过程①扫描磁盘取排序字段+指针→排序
②按排序结果回表取完整数据		一次性取出所有列 → 在 buffer 排序 → 直接输出
磁盘扫描2次1次
内存占用小（只取排序列+指针）大（取全部列）
风险较慢但稳定数据超过 sort_buffer 时反而加剧 IO
优化策略：
① 增大 sort_buffer_size（默认 1MB，建议 1M-8M）
② 增大 max_length_for_sort_data（默认 1024 字节，提高单路排序概率）
③ 禁止 SELECT * — ORDER BY 时只查需要的列
📦 GROUP BY 优化

GROUP BY 先排序后分组，索引原则几乎跟 ORDER BY 一致。额外注意：

WHERE 高于 HAVING — 能在 WHERE 里限定的条件就不要放 HAVING 中。
GROUP BY 即使没有过滤条件也能用上索引（这点和 ORDER BY 不同）。

十、慢查询日志与工具

🐌 慢查询配置
-- 查看慢查询开关（默认OFF）
SHOW VARIABLES LIKE '%slow_query_log%';

-- 临时开启（重启失效）
SET GLOBAL slow_query_log=1;

-- 设置阈值（大于该值才记录，非>=）
SET GLOBAL long_query_time=3;

-- 查看慢查询计数
SHOW GLOBAL STATUS LIKE '%Slow_queries%';

-- 永久生效 /etc/my.cnf [mysqld]
slow_query_log = 1
slow_query_log_file = /var/lib/mysql/localhost-slow.log
long_query_time = 3
log_output = FILE
🔧 mysqldumpslow 分析工具
# 常用命令示例
mysqldumpslow -s r -t 10 /var/lib/mysql/localhost-slow.log
# -s 排序方式：c(访问次数) l(锁时间) r(返回记录) t(查询时间)
# -t 返回前N条

mysqldumpslow -s at -t 5 -g 'left join' /var/lib/mysql/localhost-slow.log
# -g 正则过滤，大小写不敏感
👁️ View 视图
-- 创建视图（封装复杂SQL，提高复用性）
CREATE VIEW v_emp_dept AS
SELECT e.name, d.deptname
FROM emp e LEFT JOIN dept d ON e.deptid = d.id;

-- 使用视图（和普通表一样）
SELECT * FROM v_emp_dept WHERE deptname='研发部';
视图只保存 SQL 逻辑，不保存查询结果，数据随原表动态变化。适用于共用查询结果和报表场景。

十一、主从复制

🔄 复制原理
⚙️ 一主一从配置
--- 主机 my.cnf ---
[mysqld]
server-id=1
log-bin=mysql-bin
binlog_format=MIXED          # STATEMENT/ROW/MIXED

--- 从机 my.cnf ---
[mysqld]
server-id=2
relay-log=mysql-relay

--- 主机上创建复制账户 ---
GRANT REPLICATION SLAVE ON *.* TO 'slave'@'%' IDENTIFIED BY '123456';
FLUSH PRIVILEGES;

--- 查看主机状态（记下 File 和 Position）---
SHOW MASTER STATUS;

--- 从机上配置复制 ---
CHANGE MASTER TO
  MASTER_HOST='192.168.1.100',
  MASTER_USER='slave',
  MASTER_PASSWORD='123456',
  MASTER_LOG_FILE='mysql-bin.000001',
  MASTER_LOG_POS=154;

START SLAVE;
SHOW SLAVE STATUS \G;   -- 检查 Slave_IO_Running 和 Slave_SQL_Running
🔧 常见排障
Slave_IO_Running: No 排查步骤：
① STOP SLAVE; START SLAVE; 重启复制
② 检查主从防火墙是否关闭，是否能 ping 通
③ 确认 CHANGE MASTER TO 语句中的 IP/用户/密码/File/Position 是否正确
④ 检查 /var/lib/mysql/auto.cnf 中 uuid 是否一致（主从不能相同）
重置主从：从机执行 STOP SLAVE; RESET SLAVE;（清空 relay log），主机执行 RESET MASTER;（清空 binlog，重新初始化）。
